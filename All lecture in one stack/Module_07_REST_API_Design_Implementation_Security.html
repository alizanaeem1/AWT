7.1 — API & REST (Introduction)

Designing REST API — PDF flow

📡
API (Application Programming Interface)

Woh rules jo different software systems ko aapas me baat karwate hain — client server communication bridge.

Real-life: Foodpanda / Careem — app server ko request bhejti hai (“available restaurants”), server JSON response deta hai.

📐
REST (Representational State Transfer)

Networked applications design karne ka architectural style — HTTP protocol, standard methods, stateless communication.

REST principles (short): Stateless • Client–Server • Uniform Interface • Resource-based (detail 7.2 me six principles bhi).

7.2 — Six REST Principles

Implementing lecture

① Client–Server

Frontend / backend alag — scale independently.

② Stateless

Har request complete context; server client session memory pe depend nahi.

③ Cacheable

Responses cache ho sakti hain — performance.

④ Uniform Interface

URLs + HTTP methods consistent.

⑤ Layered System

Security, DB layers — client ko transparent.

⑥ Code on Demand

Optional — rarely emphasized in basic REST APIs.

💬
Discussion

Stateless kyu? • Session server pe store ho to scaling issues • GET se sensitive data fetch kyu avoid — caching/bookmarks/logging risks.

Express — Student Management CRUD

7.2 step-by-step

Resource: Student — fields id, name, email, department. Base path: /api/students.

server.js — setup + data
const express = require('express');
const app = express();
app.use(express.json());
app.listen(3000, () => console.log('Server running on port 3000'));

let students = [
  { id: 1, name: 'Ali', email: 'ali@gmail.com', department: 'CS' },
  { id: 2, name: 'Sara', email: 'sara@gmail.com', department: 'IT' }
];
CRUD routes
app.get('/api/students', (req, res) => res.json(students));

app.get('/api/students/:id', (req, res) => {
  const student = students.find(s => s.id == req.params.id);
  if (!student) return res.status(404).json({ message: 'Student not found' });
  res.json(student);
});

app.post('/api/students', (req, res) => {
  const newStudent = { id: students.length + 1, ...req.body };
  students.push(newStudent);
  res.status(201).json(newStudent);
});

app.put('/api/students/:id', (req, res) => {
  const student = students.find(s => s.id == req.params.id);
  if (!student) return res.status(404).json({ message: 'Student not found' });
  student.name = req.body.name;
  student.email = req.body.email;
  student.department = req.body.department;
  res.json(student);
});

app.delete('/api/students/:id', (req, res) => {
  students = students.filter(s => s.id != req.params.id);
  res.json({ message: 'Student deleted successfully' });
});

Bookstore /books routes same pattern — sirf resource name badlo.

7.3 — Why REST API Security?

Overview PDF

Agar APIs secure na hon: data theft • server overload • unauthorized access — e.g. banking API without authentication = disaster. REST methods GET POST PUT DELETE — security layer alag se design karni parti hai.

🌍

CORS (Cross-Origin Resource Sharing)

Browser-enforced

CORS control karta hai ke dusre domain se resources kaise request ho — Same-Origin Policy ke sath. Example: frontend http://example.com vs API http://api.example.com different origins.

Response headers (concept)
Access-Control-Allow-Origin: https://example.com
Access-Control-Allow-Methods: GET, POST
Access-Control-Allow-Headers: Content-Type
⛔
Sensitive APIs

Access-Control-Allow-Origin: * avoid — sirf trusted domains • methods restrict • tokens server-side verify.

⚠️

XSS & DDoS

7.3 PDF

XSS

Malicious script injection — stored / reflected / DOM-based. Impact: cookies, session hijack, redirects.

Mitigation: input validation, output encoding, HttpOnly cookies, CSP.

DDoS

Traffic flood → availability loss. Example ref: 2016 Dyn incident (Twitter, Netflix, GitHub affected).

Mitigation: rate limiting, CAPTCHA, API gateway, load balancers, CDN.

⚖️
Targets

XSS often users / browser context • DDoS servers / availability.

Authentication vs Authorization

AuthN • AuthZ

AuthenticationAuthorization
MeansKaun ho?Kya kar sakte ho?
ExamplesPassword, OTP, biometricsStudent vs admin routes
AnalogyUniversity ID checkLab access permission

Common REST methods list: Basic auth • API keys • JWT • OAuth 2.0 (“Login with Google”). JWT: compact token — Header.Payload.Signature • Authorization: Bearer <token> (RFC 7519).

🔄

Session Handling in REST

Traditional vs token-based

⚡
Tension

REST stateless philosophy • phir bhi user state chahiye — is liye patterns align carefully.

Session-based

Server session store • cookie me session id — scaling / microservices me mushkil.

Token-based (JWT)

Server token issue • client har request me bhejta hai • scalable • mobile-friendly.

  • HTTPS • HttpOnly / Secure cookies jab cookies use hon
  • Short expiry • refresh tokens • revoke compromised tokens
  • Flow: login → validate → JWT → header me verify → allow/deny
