Authentication vs Authorization — Fark samjho

Yeh do alag cheezein hain — confuse mat ho!

Pehle in do concepts ko clearly samajh lo. Bahut log inhe ek hi cheez samajhte hain — lekin yeh bilkul alag hain!

🔑
Authentication — "Tum kaun ho?"
Yeh verify karta hai ke user wahi hai jo woh kehta hai. Jaise Facebook pe login karna — username aur password dena. Server confirm karta hai ke "haan, yeh banda registered hai."
🔓
Authorization — "Tum kya kar sakte ho?"
Yeh decide karta hai ke logged-in user ko kya access hai. Admin sab kuch delete kar sakta hai, lekin normal user sirf apni posts dekh sakta hai.
🏨
Real Life Analogy
Hotel mein check-in karo (Authentication). Tumhara room card sirf tumhara room kholega, manager ka room nahi (Authorization). Dono alag processes hain!

Session-based Auth — Purana tarika aur uski problems

Pehle session-based authentication use hoti thi — server har logged-in user ka data apni memory mein rakhta tha. Yeh chhoti apps ke liye theek tha, lekin...

⚠️
PROBLEM — Session ki Limitations
Jab lakhon users hon, server ki memory full ho jaati hai. Agar multiple servers hain (load balancing), toh ek server dusre server ki session nahi jaanta. Yeh distributed systems mein kaam nahi karta!
02

JWT kya hai? — Movie Ticket Analogy

JSON Web Token — compact, secure, stateless

JWT (JSON Web Token) ek special token hai jo client aur server ke beech mein secure information bhejne ka kaam karta hai. Yeh digitally signed hota hai — matlab koi isko secretly change nahi kar sakta.

🎟️
Movie Ticket Analogy
JWT bilkul movie ticket jaisi hai. Ticket kharido (login karo) → ticket dekho aur andar jao (token bhejo) → cinema ko baar baar verify karne ki zaroorat nahi! Ticket khud proof hai.
📦
Self-Contained hai
JWT ke andar user ki sari zaroorat ki info hoti hai — ID, role, expiry time. Server ko database check karne ki zaroorat nahi. Token khud sab kuch bata deta hai!
🌐
Stateless hai
Server kuch bhi store nahi karta. Har request ke saath token aata hai, server sirf verify karta hai. Yahi wajah hai ke yeh large distributed systems mein perfect hai.

REST APIs mein JWT kyun use karte hain?

⚖️
Stateless = REST ke saath Perfect
REST APIs by design stateless hoti hain — yani server koi state nahi rakhta. JWT bhi stateless hai, toh dono ek saath perfect kaam karte hain!
📈
Scalability
Ek server pe login karo, doosre server pe bhi kaam karta hai. Server ke paas session store nahi hoti. Jitne servers chaaho add karo — sab kaam karega!
🔗
Cross-Domain kaam karta hai
Cookies sirf usi domain pe kaam karti hain. JWT kisi bhi domain pe bhej sakte ho Authorization header mein. Mobile apps, web apps, sab ke liye!
🎓
EXAM TIP
JWT ko 3 words mein yaad karo: Compact (chhota size), Self-contained (khud mein sari info), Stateless (server kuch store nahi karta). Exam mein yahi poochha jaata hai!
03

JWT Structure — 3 Parts

Header.Payload.Signature — teen hisson ka secret

JWT ke 3 parts hote hain, ek dot (.) se alag hote hain. Har part Base64 encoded hota hai. Neeche yeh visual dekho:

🔵 Header
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9
.
🟡 Payload
eyJ1c2VySWQiOjEwMSwicm9sZSI6ImFkbWluIiwiZXhwIjoxNzEyMzQ1Njc4fQ
.
🟢 Signature
SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

Upar diye token ke 3 parts hain — click karke dekho! Yeh ek actual JWT ka format hai.

Part 1: Header

Header mein sirf do cheezein hoti hain — token ka type (JWT) aur jo algorithm use ho raha hai signature banane ke liye.

JSON — Header
{
  "alg": "HS256",   // HMAC SHA-256 algorithm
  "typ": "JWT"     // Token type
}

// Phir yeh Base64 encode hota hai:
// eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9

Part 2: Payload (Claims)

Payload mein actual data hota hai — claims. User ki ID, role, aur expiry time. Yeh public info hai — koi bhi Base64 decode karke padh sakta hai, isliye yahan sensitive data mat rakho!

JSON — Payload
{
  "userId": 101,          // User ki ID
  "role": "admin",        // User ka role
  "exp": 1712345678,      // Expiry timestamp
  "iat": 1712259278       // Issued at (kab bana)
}

// ⚠️ YAAD RAKHO: Payload Base64 encoded hai, encrypted NAHI!
// Password, credit card — yahan BILKUL mat rakho!
🚨
SECURITY WARNING — Payload Readable Hai!
JWT ka Payload encrypted nahi hota — sirf Base64 encode hota hai. Koi bhi isko decode kar sakta hai aur andar ki info parh sakta hai. Isliye kabhi bhi password, credit card number, ya private info payload mein mat rakho!

Part 3: Signature

Signature sabse important part hai. Yeh ensure karta hai ke token ke saath koi tamper nahi hua. Agar koi ek letter bhi change kare, signature invalid ho jaata hai!

Signature Formula
// Signature aise banta hai:
HMACSHA256(
  Base64Encode(header) + "." + Base64Encode(payload),
  "mera-secret-key-jo-sirf-server-ko-pata-hai"
)

// Agar koi payload change kare → signature match nahi karega
// Server foran detect kar leta hai ke token fake hai!

JWT Auth Flow — Step by Step

Login se le kar protected route tak — poora safar

Ab samjhenge ke JWT actually REST API mein kaise kaam karta hai — login se shuru hokar har request pe verification tak.

1
👤 User login karta hai — POST /login ke saath username aur password bhejta hai
↓
2
🔍 Server credentials verify karta hai — database mein check karta hai ke user exist karta hai aur password sahi hai
↓
3
🏭 Server JWT generate karta hai — secret key se sign karta hai aur client ko bhejta hai
↓
4
💾 Client token store karta hai — LocalStorage ya Cookies mein rakha jaata hai
↓
5
📡 Client protected route pe request karta hai — Authorization header mein token bhejta hai: Bearer <token>
↓
6
✅ Server token validate karta hai — signature check, expiry check, phir data return karta hai

Code Example — Node.js mein JWT

JavaScript — Login + Token Generate
const jwt = require('jsonwebtoken');

// Step 1: Login route
app.post('/login', (req, res) => {
  const { username, password } = req.body;

  // Step 2: Credentials verify karo (database se)
  const user = findUser(username, password);
  if (!user) return res.status(401).send('Wrong credentials!');

  // Step 3: JWT generate karo
  const token = jwt.sign(
    { userId: user.id, role: user.role },  // payload
    'mera-secret-key',                      // secret
    { expiresIn: '24h' }                   // expiry
  );

  res.json({ token }); // token client ko bhejo
});
JavaScript — Token Verify (Middleware)
// Middleware jo har protected route se pehle chalega
const verifyToken = (req, res, next) => {
  const authHeader = req.headers['authorization'];
  const token = authHeader && authHeader.split(' ')[1]; // "Bearer TOKEN"

  if (!token) return res.status(401).send('Token nahi mila!');

  try {
    const decoded = jwt.verify(token, 'mera-secret-key');
    req.user = decoded; // user info aage bhejo
    next(); // sahi hai, aage jao
  } catch (err) {
    res.status(403).send('Token invalid ya expire ho gaya!');
  }
};

// Protected route pe middleware use karo
app.get('/dashboard', verifyToken, (req, res) => {
  res.json({ message: `Welcome, User ${req.user.userId}!` });
});
💡
TIP — University ID Card
JWT ko University ID card samjho. Pehle admission liya (register), ID card mili (token). Library mein ID dikhao (token bhejo), woh check karte hain ke valid hai (verify). Har dafa naya register nahi karna padta!

Security & Best Practices — Safe Raho

Advantages, limitations aur security tips

✅
Advantages of JWT
Stateless — server kuch store nahi karta
Scalable — multiple servers pe kaam karta hai
Fast — database hit nahi hoti har request pe
Cross-domain — mobile, web sab jagah
❌
Limitations of JWT
Revoke nahi kar sakte — token expire hone se pehle band nahi hota
Size bada ho sakta hai — har request mein bhejni padti hai
Security risk — galat jagah store kiya toh problem

🛡️ 5 Security Best Practices

1
🔒 HTTPS use karo hamesha — HTTP pe token bheja toh koi bhi beech mein pakad sakta hai (Man-in-the-Middle Attack)
2
⏳ Expiry time daal — Token hamesha expire hona chahiye. { expiresIn: '1h' } — zyada lamba time nahi rakhna
3
🔑 Strong secret key use karo — "abc123" nahi chalega! Long, random secret key use karo aur isko environment variable mein rakho
4
🚫 Sensitive data payload mein mat rakho — Password, SSN, credit card — yeh sab readable hain. Sirf user ID aur role rakho
5
🔄 Refresh Tokens use karo — Short-lived access token + long-lived refresh token. Security aur convenience dono milen

🎓
EXAM TIP — Revocation Problem
Exam mein JWT ki sabse badi limitation yeh hai: ek baar token issue ho jaaye, expire hone se pehle usse revoke nahi kar sakte (jab tak server-side blacklist na rakho). Yahi wajah hai ke tokens ka expiry time chhota rakhte hain aur Refresh Token pattern use karte hain.
